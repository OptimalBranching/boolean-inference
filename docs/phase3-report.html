Phase 2 answered "what is forced." Phase 3 answers the harder question a search has to make at every step: given the clauses and a chosen variable, which local neighborhood does the solver look at, which assignments there are even possible, and what split should it branch on? Below: tiny instances, the real numbers the engine prints, and why each is the answer you can check by hand.

CNF + focus var→ ① grow a region→ ② contract it→ ③ optimal branching rule

Part A — growing a region ① locality

The solver doesn't reason over the whole formula at once. It picks a focus variable and walks outward k hops — variable → the clauses touching it → their other variables — collecting a small local neighborhood. Larger k = wider window.

1A local window that widens with k
in · clauses
(x1∨x2) ∧ (x2∨x3) ∧ (x3∨x4) ∧ (x4∨x5)
in · focus
x3
out · k=1
region vars {x2, x3, x4} from clauses [#1, #2]
out · k=2
region vars {x1, x2, x3, x4, x5} from clauses [#0, #1, #2, #3]
why
At k=1 the engine grabs only the clauses that mention x3 — (x2∨x3) and (x3∨x4) — and their variables {x2,x3,x4}. At k=2 it takes one more hop, pulling in the clauses on x2 and x4, so the window reaches the whole chain. This is how the solver keeps each decision cheap: it works on a bounded slice, not the entire formula.

Part B — contracting the region ② what's possible

Once a region is chosen, the engine contracts it: it computes exactly which assignments to the region's variables have at least one satisfying completion of the rest. Impossible local patterns are eliminated up front — before any branching happens.

28 patterns collapse to the 3 that are consistent
in · clauses
(¬x1∨x2) ∧ (¬x2∨x3) ∧ (¬x1∨¬x3)
in · focus
x1  (region grows to {x1, x2, x3})
out
3 of 8 patterns survive: x1=0 x2=0 x3=0 x1=0 x2=0 x3=1 x1=0 x2=1 x3=1
why
There are 2³ = 8 ways to assign (x1,x2,x3). The contraction keeps only those obeying all three clauses. Check the survivors: each has x1=0 — because x1=1 would force x2=1 (clause 1) and forbid x3 (clause 3) while x2=1 forces x3=1 (clause 2), a contradiction. So x1 is effectively pinned to 0 locally, and only 3 patterns remain. The engine discovered this by contraction alone, with no guessing.

Part C — the branching rule ③ what the search tries

This is the payoff. From the surviving local patterns the engine builds the optimal branching rule: a small set of mutually-exclusive cases that together cover every possibility — chosen (by set cover) to make the search tree as shallow as possible. This is exactly what branch-and-reduce executes at each node.

3XOR forces a clean 2-way split
in · clauses
(x1∨x2) ∧ (¬x1∨¬x2)   — i.e. x1 ⊕ x2
in · focus
x1  (region {x1, x2})
out · rule
branch 2 ways: branch 1: x1=1 ∧ x2=0 branch 2: x1=0 ∧ x2=1
why
XOR means the two variables must differ, so only two local patterns survive: (1,0) and (0,1). The rule branches on exactly those two — exhaustive (every solution falls in one) and mutually exclusive (no overlap). Note each branch fixes both variables at once: choosing x1 immediately determines x2, so the engine commits both in one step instead of two.
4A neighborhood that implies a forced literal — branch count drops to 1
in · clauses
(x1∨x2∨x3) ∧ (¬x1∨x2) ∧ (¬x2∨x3)
in · focus
x2  (region {x1, x2, x3})
out · measure
3 unfixed vars at this node
out · rule
branch 1 way: branch 1: x3=1
why
The set-cover optimizer found that every surviving local pattern has x3=1: if x3 were 0, clause 3 forces x2=0, clause 2 then forces x1=0, and clause 1 collapses to false. So instead of a real branch, the "rule" is a single forced literal x3=1 — a free reduction (branching factor γ=1) that shrinks the problem without splitting the search at all. Recognizing these is precisely what makes the tree small.
How to read a "branch": each branch is a conjunction of literals the search would assume together, then propagate (Phase 2) and recurse on. A rule with k branches splits the node k ways; k=1 means no split — a forced simplification. The engine never returns an unsound rule: the branches always cover every locally-feasible assignment.

What this demonstrates

Across these instances the Phase 3 engine: ① grows a bounded local region around a chosen variable (cheap, k-hop), ② contracts it to keep only the assignments that are actually possible (8→3, pruning before any search), and ③ turns those into an optimal branching rule — a 2-way exhaustive split for XOR, and a single forced literal when the neighborhood implies one. That region → contraction → rule pipeline is the decision the branch-and-reduce search (Phase 4) will make at every node.
